## Контекст проекта

**Домен**: backend для интернет-магазина пиццы и напитков.  
**Стек**: Symfony или Laravel, PostgreSQL, RabbitMQ, Redis, Docker.  
**Цель**: отработать построение масштабируемого API и продемонстрировать инженерную культуру.

### Бизнес-видение

- **Пользователи и роли**: гость, покупатель, администратор. Гость просматривает каталог, покупатель управляет корзиной и
  оформляет заказы, администратор модерирует контент и управляет заказами.
- **Регистрация**: имя, телефон и email обязательны. После успешной регистрации пользователю отправляется приветственное SMS
  (через заглушку).
- **Каталог**: на старте две категории ― пицца и напитки. Продукты имеют базовые атрибуты (название, описание, цена, вес,
  категория) и расширяемые характеристики (КБЖУ, острота, признак вегетарианского блюда и т. п.). Важно заложить гибкость,
  позволяющую добавлять новые свойства без миграции схемы каждые две недели.
- **Корзина**: одна корзина на пользователя, максимум 10 пицц и 20 напитков одновременно. Неавторизованный пользователь видит
  только каталог.
- **Оформление заказа**: для оформления требуется минимум один товар в корзине, выбор способа доставки (самовывоз или курьер) и
  заполненный адрес для курьерской доставки. Адрес включает: область, город, улица, дом, подъезд, квартира, индекс. В заказе не может быть более 20 позиций. Статусы заказа: `created`, `paid`,
  `in_progress`, `delivering`, `completed`, `cancelled`. Обновление статуса доступно администратору и внешней системе, которая
  отправляет событие.
- **Уведомления**: при создании заказа и каждом изменении статуса пользователю отправляется email-уведомление (через заглушку).
- **Отчётность**: ежедневный scheduler инициирует генерацию отчёта о продажах, присваивает отчёту идентификатор и после
  завершения публикует событие `report.completed` в RabbitMQ. Готовый отчёт сохраняется в локальный `.jsonl` файл; каждая строка
  отражает факт продажи (товар повторяется столько раз, сколько был куплен).

Пример строки отчёта:

```jsonl
{"product_name": "Маргарита", "price": 500, "amount":1, "user": {"id":1}}
{"product_name": "Пепперони", "price": 650, "amount":1, "user": {"id":2}}
```

## Формат выполнения

- Репозиторий на GitLab с историей коммитов (логичные, атомарные изменения). Делаем отдельный МР на каждый этап
- README с инструкциями по запуску, самим заданием
- Набор сидеров/фикстур
- Каждая задача (кроме первой) делается в отдельной ветке
- Формат ветки - feature/task-2/some-name
- На ревью присылай мне MR
- В MR 
  - указать себя как asignee, 
  - меня как reviewer, 
  - затрекать время приблизительно ушедшее на задачу, 
  - добавить описание таски
- Треды которые открываю я, как ревьюер, я же должен резолвить. 
- Для удобства ты можешь помечать уже исправленные замечания каким-либо смайликом
- Время на задачах указано очень приблизительное, будет со временем калиброваться, не обращай много внимания на него, так как зависит от человека и фреймворка

## 🐳 Этап 1: Docker окружение
**Время**: 2-3 часа

### Задачи:
- Инициализировать проект (Symfony/Laravel)*
- Создать docker-compose.yml с сервисами:*
  - PHP-FPM*
  - PostgreSQL*
  - Nginx*
- Настроить multistage image для PHP для dev и prod окружений*
- разделить .env для infrastructure и application. лучше вынести приложение все в отдельную папку на уровень ниже*
- проверить, что порты в docker-compose.yaml указывающие на хост машину не захардкожены и можно менять через енв, дабы обеспечить легкое разворачивание на других машинах (у других разработчиков к примеру могут быть заняты определенные порты)*
- проверить, что для всех образов указаны версии, чтобы не допустить возникновения сложнонаходимых багов из-за того что разные версии условной посгри стянулись на разных машинах*
- проверить, что на базе висит healthcheck, а сервисы имеют зависимости друг от друга*
- Настроить volumes для кода и данных БД*
- Проверить, что приложение доступно через браузер*
- Иметь README.md с шагами развертывания*
- сделать себе Makefile/Taskfile*
- отдельные файлы - php.ini, xdebug.ini, php-fpm.conf, прокинутые в образ, с минимальными параметрами*
- использовать mlocati для установки зависимостей*
- проверить, что зависимости composer ставятся по разному в зависимости от образа*
- добавить docker-compose.prod.yaml с оверрайдами под прод*

### Чему научимся:
- Поднимать изолированное окружение через Docker
- Писать docker-compose.yml
- Конфигурировать веб-сервер
- Понимать архитектуру микросервисов

### Результат:
Работающее окружение в контейнерах, доступное на localhost

## 🎯 Этап 2: CRUD продуктов
**Время**: 5 часов

### Задачи:

- Определись с архитектурой. Я рекомендую использовать modulith + package-by-layer как самую распространенную, 
  но можно clean/ddd/package-by-feature использовать (они ортогональны)*
- Создай сущность Product с базовыми полями (name, description, price, weight, category)*
- Создай миграции БД*
- Реализуй CRUD endpoints:*
  - GET - список продуктов*
  - GET - один продукт*
  - POST - создание*
  - PATCH - обновление*
  - DELETE - удаление*
- Добавь валидацию входящих данных*
- сваггер пока не надо, позже под него будет опциональный блок
- убедись, что не используешь магические числа/строки лишний раз, например 201 (статус ответа), а предпочитаешь константы *
  (для кодов ответов например есть Response из HttpFoundation)
- убедись, что ты используешь пагинацию при чтении списка ресурса, во избежание исчерпывания оперативки*
- для симфы - убедись что у тебя не зашкаливает анемичность модели до той степени, что нет даже конструктора с обязательными полями*
- Проверь свой проект на наличие ненужных файлов, например в ларе это может быть `vite.config.json`. *
  Удали ненужные файлы, чтоб не создавалось ощущение магии в проекте
- проверь, что .idea не закоммичена
- для лары - поставить [пакет](https://github.com/barryvdh/laravel-ide-helper),*
  сгенерить докблоки в классы моделей непосредственно для улучшения автокомплита и статанализа*
- используй ресурсы для лары*

### Чему научимся:
- Работать с миграциями БД
- Строить RESTful API
- Работать с ORM (Doctrine/Eloquent)
- Валидировать входящие данные (validation rules)

## 🐰 Этап 8: RabbitMQ + Отчеты
**Время**: 5-6 часов

### Задачи:
- Добавить RabbitMQ в Docker Compose
- Настроить exchanges и queues:
  - `reports.generate` - запросы на генерацию
  - `reports.completed` - результаты генерации
- Создать команду генерации отчета:
  - Читает заказы за период
  - Генерирует JSONL файл с проданными товарами
  - Публикует событие о завершении
- Настроить Symfony Scheduler или Laravel Schedule:
  - Ежедневная генерация в 02:00
  - Альтернатива через RabbitMQ message
- API для отчетов:
  - `POST /api/reports` - запустить генерацию
  - `GET /api/reports/{id}` - статус отчета
  - `GET /api/reports/{id}/download` - скачать файл
- Написать тесты для async обработки
- Файлы хранить в MinIO

Пример строки отчёта:

```jsonl
{"product_name": "Маргарита", "price": 500, "amount":1, "user": {"id":1}}
{"product_name": "Пепперони", "price": 650, "amount":1, "user": {"id":2}}
```

### Чему научимся:
- Работать с очередями сообщений
- Асинхронная обработка задач
- Background jobs и workers
- Scheduler для периодических задач
- S3 object storage

### Результат:
Асинхронная система генерации отчетов

## ❓ FAQ

### Общие вопросы

**Q: Можно использовать готовые бандлы/пакеты?**  
A: Да, для JWT, Queue, Cache приветствуется использование проверенных решений (LexikJWTAuthenticationBundle, Symfony Messenger, Laravel Queue и т.д.).

**Q: Нужно делать frontend?**  
A: Нет, только API. Можно использовать Postman/Swagger UI для тестирования.

**Q: Обязательно все этапы?**  
A: Нет, смотрите таблицу критичности. Минимум для прохождения менторства - этапы 1-6. Остальные этапы прорабатываются по мере обучения.

**Q: Сколько времени на выполнение?**  
A: Проект рассчитан на 2-3 недели работы с ментором, где каждый этап обсуждается и ревьюится.

### Технические вопросы

**Q: Что делать если что-то непонятно?**  
A: Напишите мне. Часть неясностей специально оставлена для обсуждения архитектурных решений.

**Q: Как часто делать коммиты?**  
A: Коммиты должны быть атомарными и логичными. Один коммит = одна завершенная задача. Используйте conventional commits (feat:, fix:, refactor: и т.д.).
